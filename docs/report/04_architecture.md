## 2 Проектная часть — архитектура приложения

### 2.1 Архитектурный подход (слои)
Для проекта выбирается трёхслойная архитектура с однонаправленными зависимостями,
обеспечивающая тестируемость и поддерживаемость (NFR-06):

- **UI (presentation)**: экраны Flet, компоненты, обработка событий.
- **Domain**: правила и модели предметной области (валидации, расчёты агрегатов).
- **Infrastructure**: SQLite-репозитории, схема БД, шифрование.

Зависимости направлены «внутрь»: `ui_flet → core ← infra`.
UI-слой не импортирует инфраструктуру напрямую; доступ к данным — только через репозитории.

### 2.2 Структура модулей
```
src/
  core/                      # доменная логика — нет внешних зависимостей
    models.py                # Transaction, TransactionType (frozen dataclass)
    reporting.py             # totals_for_period, expense_by_category,
                             #   expense_by_day, income_by_day
    perf.py                  # бенчмарк-хелперы
  infra/
    db/
      connection.py          # connect() + transaction() context manager
      schema.py              # init_schema(), SCHEMA_VERSION = 3
      repositories.py        # CategoryRepository, TransactionRepository,
                             #   GoalRepository, ReminderRepository
    security/
      crypto.py              # AES-256-GCM + scrypt KDF
  ui_flet/                   # UI-слой (Flet)
    theme.py                 # цвета, пути к БД, card_shadow()
    state.py                 # Repos dataclass
    components.py            # переиспользуемые виджеты: build_sidebar,
                             #   metric_card, tx_row, empty_state и др.
    screens/
      overview.py            # главный экран: метрики, график, цели, операции
      operations.py          # список операций с навигацией по месяцам
      goals.py               # финансовые цели
      reminders.py           # напоминания
      reports.py             # аналитика: расходы по категориям и дням
      categories.py          # управление категориями
      settings.py            # настройки: смена пароля, о приложении

main_flet.py                 # точка входа: роутинг, диалог пароля,
                             #   шифрование БД на старте и при закрытии
```

### 2.3 Основные экраны приложения
- **Главная**: сводка за месяц (баланс, доходы, расходы), динамика на графике,
  топ расходов по категориям, последние операции, цели.
- **Операции**: список транзакций с навигацией по месяцам, фильтр по типу,
  добавление и удаление.
- **Аналитика**: расходы по категориям (ProgressBar-ряды), расходы по дням.
- **Цели**: создание/редактирование/удаление целей, прогресс, пополнение.
- **Напоминания**: список с подсветкой просроченных, создание/удаление.
- **Категории**: CRUD категорий с выбором типа (доход/расход/оба).
- **Настройки**: смена пароля, статус шифрования, информация о приложении.

### 2.4 Хранение данных
- Локальная **SQLite** БД (один файл).
- Доступ к БД через репозитории (CRUD + запросы агрегации для отчётов).
- `isolation_level=None` (autocommit) + `journal_mode=DELETE` (без WAL-файлов).
- Суммы хранятся в копейках (`amount_cents: int`) — никаких float.
- Даты хранятся как ISO-8601 `TEXT`.

### 2.5 Безопасность (на уровне архитектуры)
Шифрование — обязательный компонент (FR-07, NFR-05):
- KDF: **scrypt** (n=2¹⁴, r=8, p=1) — устойчив к GPU-атакам.
- Шифрование файла: **AES-256-GCM** (конфиденциальность + целостность).
- Формат: `MAGIC(4) + salt(16) + nonce(12) + ciphertext`.
- **Desktop**: `.enc`-файл на диске; при запуске расшифровывается во временный
  plaintext-файл (системная temp-папка), при закрытии — пере-шифруется и удаляется.
- `journal_mode=DELETE` исключает утечку данных через WAL-файлы при аварийном завершении.
