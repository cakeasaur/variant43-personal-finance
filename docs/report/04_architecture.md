## 2 Проектная часть — архитектура приложения

### 2.1 Архитектурный подход (слои)
Для Kivy-проекта выбирается разделение на слои, чтобы обеспечить тестируемость и
поддерживаемость (NFR-06):

- **UI (presentation)**: экраны Kivy, виджеты, обработка событий ввода.
- **Application/Use cases**: сценарии (добавить транзакцию, построить отчёт и т.д.).
- **Domain**: правила/модели предметной области (валидации, расчёты агрегатов).
- **Infrastructure**: SQLite-репозитории, миграции, шифрование/ключи, работа с файлами.

Зависимости направлены «внутрь»: UI → use cases → domain; инфраструктура подключается
на границе (через интерфейсы/адаптеры).

### 2.2 Модули (предлагаемая структура кода)
Целевая структура `src/` (UI может выделяться в отдельные модули):
```
src/
  app.py                     # основной UI (Kivy, тёмная тема) + конфигурация путей/безопасности
  core/                      # доменная логика (модели/валидации/расчёты)
    models.py
    reporting.py
  infra/                     # инфраструктура (SQLite, миграции, crypto)
    db/
      connection.py
      schema.py
      repositories.py
    security/
      crypto.py
```

Entrypoint: `main.py` в корне проекта импортирует `PersonalFinanceApp` из `src/app.py`.

### 2.3 Основные экраны приложения
- **Транзакции**: список, фильтры, итог по периоду; кнопка «Добавить».
- **Редактор транзакции**: ввод суммы/даты/категории/заметки.
- **Категории**: список категорий, создание/редактирование.
- **Отчёты**: диаграммы по категориям и динамика по времени.
- **Цели**: список целей, создание/редактирование, прогресс.
- **Напоминания**: ближайшие платежи, создание/редактирование.
- **Настройки/Безопасность**: пароль/PIN, параметры приложения.

### 2.4 Хранение данных
- Локальная **SQLite** БД (один файл).
- Доступ к БД через репозитории (CRUD + запросы агрегации для отчётов).
- Схема БД и индексы проектируются с учётом быстрых выборок по периоду/категории.

### 2.5 Безопасность (на уровне архитектуры)
Шифрование рассматривается как обязательный компонент (FR-07, NFR-05).
Архитектурно предполагается:
- наличие «слоя безопасности», отвечающего за получение ключа из пароля/PIN (KDF);
- использование ключа для защиты локального хранилища (конкретная стратегия фиксируется в технологической части и реализации).

Практическая реализация учитывает кроссплатформенность:
- **Desktop**: файл БД хранится в зашифрованном виде; при запуске расшифровывается во временный plaintext-файл (не в `data/`), а при штатном закрытии снова шифруется.
- **Android**: для воспроизводимой сборки APK шифрование отключено, а БД хранится в приватном каталоге приложения.
