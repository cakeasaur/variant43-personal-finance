## 1 Аналитическая часть — предметная область «Персональные финансы»

### 1.1 Краткое описание предметной области
Персональные финансы — это учёт и анализ денежных потоков человека (доходы и
расходы), планирование будущих платежей и достижение финансовых целей.
Приложение предназначено для офлайн-учёта транзакций и получения наглядной
картины финансового состояния за выбранный период.

### 1.2 Целевая аудитория и контекст использования
- Пользователи, которые ведут бюджет самостоятельно и хотят контролировать траты.
- Пользователи, которым нужны напоминания о регулярных платежах.
- Пользователи, которые хотят копить на цель и видеть прогресс.

Контекст: мобильное использование, быстрый ввод расходов «на ходу», отсутствие
доступа к интернету не должно блокировать работу (офлайн-first).

### 1.3 Словарь терминов (глоссарий)
- **Транзакция**: запись о доходе или расходе с датой, суммой и категорией.
- **Категория**: классификатор транзакций (например, «Еда», «Транспорт»).
- **Период**: интервал дат, по которому строятся отчёты (день/неделя/месяц/год/произв.).
- **Отчёт/сводка**: агрегированные показатели по периоду (итог доходов/расходов, доли).
- **Финансовая цель**: накопление заданной суммы к сроку (или без срока) с прогрессом.
- **Напоминание**: будущий платёж с датой и (опционально) правилом повторения.
- **Локальная БД**: SQLite-файл с данными пользователя на устройстве.
- **Шифрование**: защита содержимого локальной БД/данных от чтения без ключа.

### 1.4 Основные сущности и их атрибуты (черновая модель данных)
1) **Transaction (Транзакция)**
- id
- type: income/expense
- amount
- currency (опционально, если поддерживается)
- occurred_at (дата/время)
- category_id
- note (комментарий)
- created_at/updated_at

2) **Category (Категория)**
- id
- name
- kind (income/expense/both — решение в проектировании)
- color/icon (опционально)
- order_index (опционально)

3) **Goal (Цель)**
- id
- title
- target_amount
- current_amount (или вычисление по связанным операциям)
- deadline (опционально)
- created_at/updated_at

4) **Reminder (Напоминание)**
- id
- title
- amount (опционально)
- due_date
- recurrence_rule (опционально: none/daily/weekly/monthly/custom)
- is_done / last_triggered_at

5) **Settings/Security (Настройки/Безопасность)**
- параметры приложения (валюта по умолчанию, формат дат)
- параметры защиты (наличие пароля/PIN, параметры KDF, salt)

### 1.5 Типовые пользовательские сценарии (Use cases — текстом)
- **UC-01**: Добавить расход (сумма, дата, категория, заметка).
- **UC-02**: Добавить доход.
- **UC-03**: Просмотреть список транзакций за период (фильтр по категории/типу).
- **UC-04**: Посмотреть диаграмму расходов по категориям за месяц.
- **UC-05**: Создать финансовую цель и отслеживать прогресс.
- **UC-06**: Создать напоминание о платеже и получить уведомление/индикатор.
- **UC-07**: Включить защиту данных (задать пароль/PIN), затем вход в приложение.
- **UC-08**: Экспорт/резервная копия (опционально, если успеваем и это не противоречит требованиям).

### 1.6 Ограничения и допущения (для фиксации в проекте)
- Работа офлайн; синхронизация с банками не входит в область проекта.
- Источник данных — ручной ввод пользователя.
- Для диаграмм используются агрегаты по транзакциям за период.
- Механизм напоминаний может быть реализован как:
  - системные уведомления (если поддерживается на целевой платформе), или
  - внутренняя «лента ближайших платежей» + индикатор в приложении.
  Конкретный вариант закрепляется в проектировании.
