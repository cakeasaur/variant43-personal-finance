## Требования к программному продукту (вариант 43)

### 3.1 Функциональные требования (FR)

#### FR-01. Ведение транзакций
- Пользователь может добавить транзакцию типа **доход** или **расход**.
- Поля транзакции: сумма, дата/время, категория, заметка (опционально).
- Пользователь может удалять транзакции.
- _Редактирование существующей транзакции — backlog (v0.2)._

#### FR-02. Категории _(backlog — не реализовано в v0.1)_
- Пользователь может создавать/редактировать/удалять категории (запланировано).
- В текущей версии доступны только предустановленные категории по умолчанию.
- Транзакция может быть создана без категории («Без категории»).

#### FR-03. Просмотр и фильтрация
- Просмотр списка транзакций за период (по умолчанию — текущий месяц).
- Фильтрация по типу (доход/расход) и по категории.
- Отображение итогов за период: сумма доходов, сумма расходов, баланс.

#### FR-04. Визуализация (диаграммы)
- Диаграмма расходов по категориям за выбранный период.
- График/диаграмма динамики (например, по дням/неделям в периоде).

#### FR-05. Финансовые цели
- Создание/редактирование/удаление цели (название, целевая сумма, срок — опционально).
- Отображение прогресса (в процентах и абсолютных значениях).

#### FR-06. Напоминания о платежах
- Создание/редактирование/удаление напоминания (название, дата, сумма — опционально).
- Поддержка повторения (минимум: без повтора / ежемесячно).
- Отображение списка ближайших напоминаний.

#### FR-07. Защита данных (шифрование локальной БД)
- Данные пользователя должны храниться локально в SQLite.
- Локальная БД (или критичные поля) должны быть защищены шифрованием.
- Пользователь должен иметь способ задать пароль/PIN для доступа к данным.

### 3.2 Нефункциональные требования (NFR)

#### NFR-01. Кроссплатформенность
- Приложение должно запускаться на Desktop (Windows, macOS) и поддерживать сборку под Android (целевой стек: Flet).

#### NFR-02. Офлайн-first
- Все функции учёта и просмотра работают без интернета.

#### NFR-03. Производительность
- Операции просмотра списка и построения отчётов должны оставаться отзывчивыми на данных порядка \(N \approx 5{,}000\) транзакций.

#### NFR-04. Надёжность и целостность данных
- Операции записи должны быть транзакционными (атомарность при добавлении/редактировании).
- Приложение корректно обрабатывает некорректный ввод (валидация суммы/дат).

#### NFR-05. Безопасность
- Данные в БД не должны читаться в открытом виде без ключа.
- Пароль/PIN не хранится в открытом виде; используется KDF (например, PBKDF2/scrypt/argon2 — выбор фиксируется в проектировании).

#### NFR-06. Поддерживаемость
- Код структурирован по слоям (UI/логика/хранилище).
- Базовый набор тестов для доменной логики и слоя БД.

### 3.3 Критерии приёмки (Acceptance criteria)
- Приложение позволяет добавить/отредактировать/удалить транзакцию, и изменения сохраняются после перезапуска.
- Есть минимум 5 категорий по умолчанию (CRUD пользовательских категорий — backlog v0.2).
- Есть отчёт за месяц: итог доходов/расходов + диаграмма по категориям.
- Можно создать цель и увидеть прогресс.
- Можно создать напоминание и увидеть его в списке ближайших; повторяемое напоминание корректно пересчитывается.
- Включение защиты (пароль/PIN) реально блокирует чтение данных без ввода секрета (демонстрируется на защите).
