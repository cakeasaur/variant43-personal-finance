## 3 Технологическая часть — обоснование выбора технологий

### 3.1 Выбор языка программирования: Python
Проект реализуется на **Python** по следующим причинам:
- высокая скорость разработки и читаемость кода;
- богатая экосистема библиотек для работы с данными, временем, тестированием;
- удобство реализации доменной логики, отчётов и тестируемых модулей;
- поддержка кроссплатформенной разработки в связке с Flet.

Целевая версия Python фиксируется как **3.12** (см. `.python-version`) для лучшей
совместимости библиотек и стабильности окружения.

### 3.2 Выбор UI-фреймворка: Flet
**Flet** выбран как основной UI-фреймворк:
- построен на Flutter-рендеринге — нативная производительность на всех платформах;
- позволяет создавать кроссплатформенные desktop- и мобильные приложения
  из единой кодовой базы на Python;
- декларативный стиль описания интерфейса, удобная система компонентов;
- официальная поддержка сборки под Windows, macOS, Android (через `flet build`).

Используемая версия: **Flet 0.85.1** + **flet-charts 0.85.1** для линейных графиков.

Для визуализации динамики (главный экран) используется **matplotlib** →
PNG-байты → `ft.Image`, поскольку `MatplotlibChart` из flet-charts работает
только в web-режиме.

### 3.3 Выбор СУБД: SQLite
**SQLite** используется как локальное хранилище, поскольку:
- обеспечивает автономную работу приложения (офлайн-first);
- не требует отдельного сервера и проста в развёртывании;
- поддерживает транзакции и индексы, что важно для целостности и скорости отчётов;
- встроена в стандартную библиотеку Python (`sqlite3`).

### 3.5 Стратегия защиты данных (шифрование локальной БД)
Требование варианта: **шифрование локальной БД** (FR-07).

#### 3.5.1 Модель угроз (упрощённо)
Защищаемся от сценария: злоумышленник получил доступ к файлу БД на устройстве
(копирование файла) и пытается прочитать данные вне приложения.

Не решаем в рамках проекта: компрометация устройства на уровне ОС, кейлоггер,
рут/джейлбрейк, аппаратные атаки.

#### 3.5.2 Подход к шифрованию
Выбрана стратегия **полного шифрования файла БД** на уровне приложения:
- пользователь задаёт пароль при первом запуске;
- из пароля выводится 256-битный ключ через **scrypt** (n=2¹⁴, r=8, p=1);
- файл SQLite оборачивается в контейнер **AES-256-GCM**;
- при запуске расшифровывается во временный plaintext-файл в системной temp-папке;
- при завершении — пере-шифровывается и plaintext удаляется.

Формат контейнера: `MAGIC(4) + salt(16) + nonce(12) + ciphertext`.

#### 3.5.3 Почему так
- Пароль/PIN не хранится в открытом виде; хранится только `salt` и параметры KDF.
- Даже при копировании файла БД, без пароля данные не читаются.
- `journal_mode=DELETE` исключает утечку данных через WAL-файлы (`-wal`, `-shm`)
  во временной директории при аварийном завершении.

#### 3.5.4 Ограничения
- При забытом пароле восстановление данных невозможно.
- Plaintext-файл существует только во время работы приложения; при штатном
  закрытии выполняется пере-шифрование (best-effort + atexit fallback).
