## 3 Технологическая часть — обоснование выбора технологий

### 3.1 Выбор языка программирования: Python
Проект реализуется на **Python** по следующим причинам:
- высокая скорость разработки и читаемость кода;
- богатая экосистема библиотек для работы с данными, временем, тестированием;
- удобство реализации доменной логики, отчётов и тестируемых модулей;
- поддержка кроссплатформенной разработки в связке с Kivy.

Целевая версия Python фиксируется как **3.12** (см. `.python-version`) для лучшей
совместимости библиотек и стабильности окружения.

### 3.2 Выбор UI-фреймворка: Kivy
**Kivy** выбран, так как:
- позволяет создавать кроссплатформенные приложения (Android/desktop);
- поддерживает декларативные интерфейсы (kv) и модульную структуру экранов;
- хорошо подходит для прототипирования и учебных проектов.

### 3.3 Выбор СУБД: SQLite
**SQLite** используется как локальное хранилище, поскольку:
- обеспечивает автономную работу приложения (офлайн-first);
- не требует отдельного сервера и проста в развёртывании;
- поддерживает транзакции и индексы, что важно для целостности и скорости отчётов.

### 3.4 Ключевые технологии и библиотеки (факт)
В проекте используются:
- **pytest** — модульные и интеграционные тесты;
- **ruff** — статический анализ и единый стиль импорта/кода;
- **pip-audit** — проверка зависимостей на известные уязвимости (SCA);
- **cryptography** — криптографические примитивы (AES-GCM, scrypt).

Визуализация отчётов реализована без внешних библиотек графиков: простые «bar chart» элементы
формируются в UI Kivy на основе агрегатов доменной логики.

### 3.5 Стратегия защиты данных (шифрование локальной БД)
Требование варианта: **шифрование локальной БД** (FR-07).

В учебном проекте важно одновременно:
1) реализовать реальную защиту данных,
2) оставить решение объяснимым и воспроизводимым для защиты.

#### 3.5.1 Модель угроз (упрощённо)
Защищаемся от сценария: злоумышленник получил доступ к файлу БД на устройстве
(копирование файла) и пытается прочитать данные вне приложения.

Не решаем в рамках проекта: компрометация устройства на уровне ОС, кейлоггер,
рут/джейлбрейк, аппаратные атаки.

#### 3.5.2 Подход к шифрованию (выбор для реализации)
Выбираем стратегию **шифрования данных приложения** с ключом, полученным из
пароля/PIN пользователя:
- пользователь задаёт пароль/PIN;
- из пароля выводится ключ через KDF (PBKDF2/scrypt/argon2 — окончательный выбор
  фиксируется в реализации);
- ключ используется для шифрования «секретного контейнера» с данными.

Выбран вариант **A) Полное шифрование файла БД** на уровне приложения:
- ключ выводится из пароля пользователя через **scrypt**;
- файл SQLite оборачивается в контейнер **AES-GCM** (конфиденциальность + целостность);
- при запуске пользователь вводит пароль, приложение расшифровывает БД во временный plaintext-файл,
  а при завершении — повторно шифрует и удаляет plaintext.

#### 3.5.3 Почему так
- Пароль/PIN не хранится в открытом виде; хранится только `salt` и параметры KDF.
- Даже при копировании файла БД, без пароля данные не читаются.

#### 3.5.4 Ограничения и риски
- При забытом пароле восстановление данных невозможно (это свойство сильного шифрования).
- Некоторые варианты «полного шифрования SQLite» требуют специфических сборок/библиотек,
  что может усложнить кроссплатформенность; этот риск будет учтён при выборе варианта A/B.

Допущение реализации: в рамках учебного проекта plaintext-файл может существовать только во время работы
приложения. При штатном закрытии выполняется пере-шифрование и удаление plaintext-файла.
